## 테스트 1: transit-encryption-mode preferred

### 개요

TLS 클러스터를 `required` → `preferred`로 온라인 전환. preferred 모드에서는 동일 포트(6379)에서 TLS/Non-TLS 양쪽 연결을 모두 수락합니다. 성능이 중요한 내부 트래픽은 Non-TLS로 연결하여 TLS 오버헤드를 회피하는 전략.

전환 명령:
```bash
aws elasticache modify-replication-group \
  --replication-group-id stviztlwuv2jozz \
  --transit-encryption-mode preferred \
  --apply-immediately
```

- TLS 연결(13.8% → 54.1% → 50.4%)보다도 훨씬 나쁨

**원인 분석:**
- preferred 모드에서 서버는 새 연결마다 첫 바이트를 읽어 TLS ClientHello인지 판별
- 대량 동시 연결 시 이 판별 과정이 직렬화되면서 Non-TLS 연결이 대기열에 갇힘
- TLS 연결은 ClientHello를 즉시 보내므로 판별이 빠르지만, Non-TLS 연결은 RESP 프로토콜 데이터를 보내기까지 지연 발생 추정

**결론:** preferred 모드는 소규모(10~50) 워크로드에서만 유효. 대규모 connection storm에서는 순수 Non-TLS 클러스터보다 훨씬 나쁜 결과를 보이므로 반드시 사전 테스트 필요.

## 테스트 2: Connection Pool + Warm-up

### 개요

매번 새 ClusterClient를 생성하는 대신, 단일 ClusterClient를 미리 생성(warm-up)하고 모든 동시 요청이 공유. TLS 핸드셰이크는 초기 warm-up 시에만 발생하고, 이후 요청은 기존 연결을 재사용.

```
[Storm 모드]  100 conns × 2 shard × 2 노드 = 최대 400회 TLS 핸드셰이크
[Pool 모드]   warm-up 시 1회 × 2 shard × 2 노드 = 4회 TLS 핸드셰이크
```

### 인사이트

**100 conns — 극적 개선 (이 테스트의 하이라이트):**
- Storm 5.3s → Pool 292ms (**18배 빠름**)
- Storm: 100개 독립 ClusterClient × 4 노드 = 최대 400회 TLS 핸드셰이크
- Pool: warm-up 시 4회 핸드셰이크 후, 100개 요청은 기존 연결 재사용
- 처리량: 19 c/s → 342 c/s (18배 향상)

**10~50 conns — 차이 없음:**
- 소규모에서는 핸드셰이크 총 횟수가 적어 병목이 되지 않음
- Pool의 warm-up 오버헤드와 Storm의 핸드셰이크 비용이 비슷한 수준

**200 conns — Pool이 오히려 느림:**
- Pool 10.4s vs Storm 5.4s
- 단일 ClusterClient의 내부 connection pool이 200개 동시 요청을 처리하면서 내부 lock contention 발생 추정
- redis crate의 ClusterClient는 내부적으로 slot별 연결을 관리하며, 동시 접근 시 mutex 경합

**500+ conns — 서버 한계 지배적:**
- 양쪽 모두 서버 측 한계로 타임아웃
- Pool이 약간 나은 성공률 (36% vs 30.2%) — 핸드셰이크 부하 감소 효과
- 2,000 conns에서는 Pool이 오히려 낮음 (36.1% vs 50.6%) — 단일 클라이언트 내부 경합

**결론:** Connection Pool은 **중규모(100~200) 동시접속에서 가장 효과적**. 대규모에서는 서버 측 한계가 지배적이며, 단일 클라이언트 내부 경합이 추가 병목이 될 수 있음.

## 테스트 4: Shard 수 증가 (2 → 4 shard)

### 개요

각 shard가 독립적으로 TLS 핸드셰이크를 처리하므로, shard를 늘리면 TLS 처리 용량이 수평 확장될 것이라는 가설 검증.

```
[2 shard] ClusterClient → 4 노드 (2M + 2R) 에 TLS 연결
[4 shard] ClusterClient → 8 노드 (4M + 4R) 에 TLS 연결  ← 핸드셰이크 2배
```

### TLS Storm (r7g.large, 4 shard, required)

|----------|-----------|------|--------|--------|
| 10 | 354ms | 10/10 | 100% | 28 c/s |
| 50 | 460ms | 50/50 | 100% | 109 c/s |
| 100 | 5.5s | 100/100 | 100% | 18 c/s |
| 200 | 15.1s | 76/200 | **38%** | 5 c/s |
| 500 | 15.4s | 160/500 | 32% | 10 c/s |
| 1,000 | 15.2s | 607/1000 | 60.7% | 40 c/s |

| 동시접속 | Wall Time | 성공 | 성공률 | 처리량 |

### 2 shard vs 4 shard 비교 (TLS Storm)

| 동시접속 | 2 shard | 4 shard | 변화 |
|----------|---------|---------|------|
| 10 | 129ms, 100% | 354ms, 100% | **2.7x 느려짐** |
| 50 | 199ms, 100% | 460ms, 100% | **2.3x 느려짐** |
| 100 | 5.3s, 100% | 5.5s, 100% | 유사 |
| 200 | 5.4s, 100% | 15.1s, 38% | **대폭 악화** |
| 500 | 30.2% | 32% | 유사 |
| 1,000 | 57.8% | 60.7% | 유사 |
| 2,000 | 50.6% | 26% | **악화** |

### 인사이트

**가설 기각 — Shard 증가는 connection storm에 역효과:**

**소규모(10~50 conns) — 2.3~2.7배 느려짐:**
- 4 shard: ClusterClient가 8개 노드 모두에 TLS 연결 → 핸드셰이크 총 횟수 2배
- 10 conns: 10 × 8 노드 = 80회 핸드셰이크 (2 shard: 10 × 4 = 40회)
- 순수 핸드셰이크 비용이 비례 증가

**200 conns — 2 shard 100% vs 4 shard 38%:**
- 2 shard: 200 × 4 노드 = 800회 핸드셰이크 → 서버 4개가 각 200회 처리
- 4 shard: 200 × 8 노드 = 1,600회 핸드셰이크 → 서버 8개가 각 200회 처리
- 서버 수는 2배이지만 핸드셰이크 총 횟수도 2배 → 순 이득 없음
- 오히려 클라이언트 측에서 8개 노드에 동시 연결하면서 리소스 경합 증가

**4 shard Pool 500 conns — 3.8% 성공 (최악):**
- Pool 모드에서도 warm-up 시 8개 노드에 연결해야 하므로 초기 비용이 높음
- 500개 동시 요청이 단일 ClusterClient를 통해 8개 노드에 분산되면서 내부 경합 극대화

**핵심 원인:**
- Cluster Mode에서 ClusterClient는 **모든 shard의 모든 노드에 연결**을 시도
- shard 증가 = 노드 수 증가 = 클라이언트당 TLS 핸드셰이크 횟수 증가
- 서버 측 TLS 처리 용량은 shard별로 독립이지만, 클라이언트 측 부하가 비례 증가하여 상쇄

**결론:** Shard 증가는 데이터 처리량 분산에는 효과적이지만, connection storm 시나리오에서는 역효과. TLS 연결 수립 성능 개선 목적으로 shard를 늘리는 것은 비권장.

### 핵심 메시지

TLS 자체를 끄는 것이 아니라, **TLS 핸드셰이크 횟수를 최소화**하는 것이 핵심:

```
[문제] 배포 시 N개 Pod × M개 shard × 2 노드 = N×M×2 TLS 핸드셰이크 동시 발생

[해결]
1. Connection Pool: 핸드셰이크를 앱 시작 시 1회로 제한 → 18x 개선
2. Staggered Reconnection: 동시 핸드셰이크 수를 시간 분산
3. Envoy Sidecar: N개 앱 연결 → 소수 TLS 연결로 다중화
```
